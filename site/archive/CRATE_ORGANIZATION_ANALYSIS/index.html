Crate Organization Analysis¶

Current Structure Overview¶

c2pa-claim-generator/
├── domain/          # Core business logic & traits (no external deps)
├── app/             # Application orchestration layer
├── infra/           # Infrastructure implementations
├── media-packager/  # Standalone media processing library
└── service-http/    # HTTP API layer (Axum)

Dependency Graph¶

service-http
    ↓
  app  ←──────┐
    ↓         │
  infra ──────┤
    ↓         │
 domain ←─────┘

media-packager (standalone, used by infra)

Strengths ✅¶

1. Clean Dependency Inversion (Ports & Adapters)¶

  • domain defines traits (Processor, JobStore, InputResolvers, OutputSink)
  • infra provides concrete implementations
  • app orchestrates without knowing implementation details
  • Follows hexagonal architecture principles well

2. media-packager as Independent Library¶

  • Zero dependency on domain/app/infra
  • Can be extracted to a separate repo easily
  • Well-documented with examples
  • Good public API design with clear exports
  • Self-contained with FFmpeg/Bento4 abstractions

3. Type-Safe Processor System¶

  • Generic Processor<Input> trait with associated types
  • Type erasure via DynProcessor for dynamic dispatch
  • Clean separation: processors in infra, registry in app

4. Feature Flags for Resolvers¶

  • infra exposes local, http, s3, etc. features
  • service-http selects only what it needs
  • Reduces binary size and dependencies

Issues & Anti-Patterns ⚠️¶

1. infra → media-packager Dependency Violation¶

Problem: 

# infra/Cargo.toml
[dependencies.media-packager]
path = "../media-packager"
- infra depends on media-packager for MediaPackagerProcessor - media-packager is supposed to be standalone - Creates coupling: changes in media-packager force infra rebuild

Impact: - Violates "library should be independent" goal - PackageAndSignProcessor in infra directly uses media-packager types

Why it happened: - MediaPackagerProcessor is a domain Processor implementation - But it wraps a library (media-packager::MediaPipeline) that knows nothing about domain

Fix Options: 1. Move MediaPackagerProcessor into media-packager itself as an adapter module 2. Create a thin adapter in infra that re-exports only what's needed 3. Keep as-is but document this as intentional (least clean)

2. Scattered Public API Definitions¶

Problem: - domain/src/lib.rs re-exports 21 items from 6 modules - No clear "these are stable, these are internal" markers - Models mixed with traits mixed with utilities

Example: 

pub use models::{
    BatchId, BatchMetrics, BatchSnapshot, ByteStream, FileResult, 
    FileStatus, JobStatus, MediaInput, OutputRef, ResolvedFileInput, 
    ResolvedFolderInput, ResolvedMediaInput, SeekableReadHandle,
};
All exported at the same level—no signal about stability.

Fix: - Group re-exports by category (traits, models, utilities) - Add #[doc(hidden)] for internal-but-public items - Consider pub(crate) for items only used within workspace

3. app Crate is Thin (Maybe Too Thin?)¶

Current State: 

// app/src/lib.rs
pub mod batch_service;
pub mod registry;
pub mod runner;
- Only 3 modules, minimal logic - Most orchestration happens in service-http/src/main.rs

Questions: 1. Should WorkerConfig, job_queue(), spawn_worker_loop() be in app or service-http? - Currently: app (good—reusable for CLI or other transports) 2. Should build_processor_registry() be in app or service-http? - Currently: service-http/src/processing.rs (bad—not reusable)

Recommendation: - Move build_processor_registry() into app as a default factory - service-http should only contain HTTP-specific code (routes, handlers, DTOs)

4. infra Public API is Flat¶

Current: 

// infra/src/lib.rs
pub use c2pa::{C2paProcessor, C2pafMP4Processor};
pub use media_packager::{MediaPackagerProcessor, PackageAndSignProcessor};
pub use resolvers::CompositeInputResolver;
pub use sinks::SimpleOutputSinkFactory;
pub use stores::{InMemoryJobStore, SqliteJobStore};

Issues: - All implementations exposed at root level - No namespace grouping (users write infra::C2paProcessor instead of infra::processors::C2pa) - Hard to see which items are "batteries included" vs. advanced

Better API: 

// Batteries-included presets
pub use presets::{default_resolvers, default_sink, default_store};

// Organized by category
pub mod processors {
    pub use crate::c2pa::{C2paProcessor, C2pafMP4Processor};
    pub use crate::media_packager::{MediaPackagerProcessor, PackageAndSignProcessor};
}
pub mod resolvers { /* ... */ }
pub mod stores { /* ... */ }
pub mod sinks { /* ... */ }

5. domain Models Could Use More Visibility Control¶

Current: 

// domain/src/lib.rs
pub mod models;  // Everything inside is pub

Problem: - models/mod.rs likely has many internal types that leak into public API - Users might depend on internal structs

Fix: - Keep models private: mod models; - Re-export only stable types: pub use models::{BatchId, BatchSnapshot, ...};

6. Missing Workspace-Level Public API Validation¶

No tracking of: - What is "stable 1.x API" vs. experimental - Breaking changes across versions - Public API surface size

Recommendation: - Add cargo-public-api to CI (already mentioned in mode instructions) - Document stability guarantees in crate-level docs

Recommendations¶

Priority 1: Fix infra → media-packager Coupling¶

Option A (Recommended): Create adapter module in media-packager 

// media-packager/src/domain_adapter.rs (feature-gated)
#[cfg(feature = "domain-integration")]
pub mod domain_adapter {
    use super::*;

    pub struct MediaPackagerProcessor { /* ... */ }

    impl domain::Processor for MediaPackagerProcessor {
        type Input = domain::ResolvedFolderInput;
        // ...
    }
}


# media-packager/Cargo.toml
[dependencies.domain]
path = "../domain"
optional = true

[features]
domain-integration = ["domain"]

Option B: Keep as-is but document - Add comment: "media-packager is workspace-internal, not published separately" - Document in README that it's a logical separation, not physical

Priority 2: Reorganize infra Public API¶

// infra/src/lib.rs
//! Infrastructure implementations for c2pa-claim-generator
//! 
//! ## Quick Start
//! ```rust
//! use infra::presets;
//! let resolvers = presets::default_input_resolvers();
//! let store = presets::in_memory_job_store();
//! ```

pub mod presets;  // Batteries-included factories

pub mod processors {
    pub use crate::c2pa::{C2paProcessor, C2pafMP4Processor};
    pub use crate::media_packager::{MediaPackagerProcessor, PackageAndSignProcessor};
}

pub mod resolvers {
    pub use crate::resolvers::CompositeInputResolver;
}

pub mod stores {
    pub use crate::stores::{InMemoryJobStore, SqliteJobStore};
}

pub mod sinks {
    pub use crate::sinks::SimpleOutputSinkFactory;
}

// Private modules
mod c2pa;
mod media_packager;

Priority 3: Move build_processor_registry() to app¶

// app/src/registry/default_registry.rs
pub fn build_default_registry() -> MapProcessorRegistry {
    // Move logic from service-http/src/processing.rs here
}

service-http just calls it: 

// service-http/src/main.rs
let processor_registry = Arc::new(app::registry::build_default_registry());

Priority 4: Add Stability Markers¶

// domain/src/lib.rs
//! # Stability Guarantees
//! 
//! ## Stable API (1.x)
//! - `Processor`, `BatchService`, `JobStore`
//! - `BatchSpec`, `InputRef`, `ProcessedArtifact`
//! 
//! ## Experimental (may change in minor versions)
//! - `ResolvedMediaInput` (pending HLS refactor)

/// Core processor trait (stable API)
pub use processor::Processor;

/// Experimental: Media input types (subject to change)
#[doc(alias = "experimental")]
pub use models::ResolvedMediaInput;

Priority 5: Audit Public Exports in domain¶

Current: 

pub use models::{
    BatchId, BatchMetrics, BatchSnapshot, ByteStream, FileResult, FileStatus, 
    JobStatus, MediaInput, OutputRef, ResolvedFileInput, ResolvedFolderInput, 
    ResolvedMediaInput, SeekableReadHandle,
};

Questions to ask: 1. Does service-http need ByteStream directly? → Probably not (internal) 2. Does external code need SeekableReadHandle? → No (implementation detail) 3. Should FileResult/FileStatus be public? → Maybe only through BatchSnapshot

Refined: 

// Public API
pub use models::{
    BatchId, BatchSnapshot, BatchSpec,
    InputRef, OutputRef, ProcessedArtifact,
};

// Re-exported for convenience (but mark as less stable)
pub use models::{
    ResolvedFileInput, ResolvedFolderInput, 
};

// Internal (used by infra/app, not external users)
pub(crate) use models::{ByteStream, SeekableReadHandle};

Architecture Maturity Assessment¶

Aspect Rating Notes
Dependency Inversion ⭐⭐⭐⭐⭐ Excellent—domain defines contracts
Separation of Concerns ⭐⭐⭐⭐☆ Good, but app is thin
Public API Clarity ⭐⭐⭐☆☆ Too many exports, no stability markers
Module Organization ⭐⭐⭐☆☆ Flat namespaces in infra
Library Independence ⭐⭐⭐☆☆ media-packager coupled to infra
Testability ⭐⭐⭐⭐☆ Traits make mocking easy
Documentation ⭐⭐⭐☆☆ Code-level docs good, crate-level sparse

Comparison to Industry Patterns¶

Similar to:¶

  • Actix Web ecosystem: Clean trait-based abstractions
  • Tokio: Separate tokio (core) from tokio-util (helpers)
  • Tower: tower (traits) + tower-http (implementations)

Different from:¶

  • Axum: More monolithic (but they're a framework, not a library)
  • Diesel: More macro-heavy, less runtime polymorphism

Final Verdict¶

Overall: 7.5/10 🎯

Strengths: - Solid hexagonal architecture foundation - Type-safe processor system with good ergonomics - media-packager is well-designed as a library

Main Gaps: 1. infra → media-packager coupling breaks library independence 2. Public API lacks organization and stability markers 3. app crate is underutilized (registry building should move there)

If you fix Priority 1-3, this becomes a 9/10 architecture—production-ready and maintainable.
