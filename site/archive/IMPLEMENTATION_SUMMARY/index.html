Implementation Summary: Package and Sign Processor¶

What Was Implemented¶

A new package_and_sign processor that combines media packaging and C2PA signing in a single operation:

  1. Packages multiple resolution MP4 files into DASH/HLS format
  2. Signs the packaged fragmented content with C2PA in place

Files Created¶

Core Implementation¶

  • infra/src/media_packager/package_and_sign_processor.rs - Main processor implementation
  • configs/media_packager_pipeline.toml - Pipeline configuration for packaging
  • configs/package_and_sign_example.json - Example request payload

Documentation¶

  • PACKAGE_AND_SIGN_PROCESSOR.md - Comprehensive processor documentation
  • CURL_COMMANDS.md - Quick reference with curl commands
  • scripts/test_package_and_sign.sh - Test script

Modified Files¶

  • infra/src/media_packager/mod.rs - Export new processor
  • infra/src/lib.rs - Export from infra crate
  • service-http/src/processing.rs - Register processor
  • media-packager/src/config.rs - Add from_toml_file method
  • media-packager/Cargo.toml - Add anyhow dependency
  • infra/src/media_packager/processor.rs - Update to use config

How to Use¶

1. Start the Server¶

cd /home/taddist/WDR/c2pa-claim-generator
RUST_LOG=info cargo run --bin service-http

2. Submit a Request (Synchronous)¶

curl -X POST http://localhost:8080/batches/sync \
  -H "Content-Type: application/json" \
  -d '{
    "inputs": [
      {
        "type": "local_folder_path",
        "path": "./test_inputs"
      }
    ],
    "processor": {
      "kind": "package_and_sign",
      "params": {
        "output_name": "my_content"
      }
    }
  }'

3. Submit Asynchronously (Submit and Poll)¶

# Submit
curl -X POST http://localhost:8080/batches \
  -H "Content-Type: application/json" \
  -d @configs/package_and_sign_example.json

# Returns: {"batch_id": "abc123..."}

# Check status
curl http://localhost:8080/batches/abc123...

Input Requirements¶

The processor expects a folder containing MP4 files: 

test_inputs/
├── video_270p.mp4
├── video_360p.mp4
├── video_540p.mp4
├── video_720p.mp4
├── video_1080p.mp4
└── audio.mp4

Output Structure¶

Creates a folder with packaged and signed content: 

<output_name>_packaged_signed/
├── stream.mpd                    # DASH manifest
├── master.m3u8                   # HLS master playlist
├── video_c2pa_video_270p_frag/
│   ├── init.mp4                  # Signed init segment
│   ├── seg_00000.m4s             # Signed fragment
│   └── ...
├── video_c2pa_video_720p_frag/
│   └── ...
└── audio_c2pa_audio_frag/
    └── ...

Parameters¶

Parameter Type Required Default Description
output_name string Yes - Name for the output
playlist_pattern string No **/playlist.m3u8 Pattern to match playlists
init_pattern string No **/init_*.mp4 Pattern to match init segments
frag_pattern string No seg_*.m4s Pattern to match fragments

Configuration Files¶

1. Pipeline Configuration (configs/media_packager_pipeline.toml)¶

Controls demuxing, fragmentation, and packaging: - Fragment duration (default: 2000ms) - Parallel operations (default: 4) - DASH profiles - HLS output enable/disable - Naming templates

2. C2PA Settings (configs/default_c2pa_sdk_settings.toml)¶

Controls C2PA signing: - Signer configuration - Manifest builder settings - Claim generation options

Key Features¶

✅ Combined Operation: Package and sign in one step
✅ In-Place Signing: No separate signed folder created
✅ Parallel Processing: Fast demuxing and fragmentation
✅ Async Support: Submit and poll or wait for completion
✅ Comprehensive Logging: Track progress with RUST_LOG
✅ Error Handling: Detailed error messages
✅ Configurable: TOML-based configuration

Endpoints¶

The processor uses existing batch endpoints:

  • POST /batches - Submit async job, returns batch_id
  • POST /batches/sync - Submit sync job, waits for completion
  • GET /batches/{id} - Check status of a batch
  • GET /health - Health check

Testing¶

Run the test script: 

chmod +x scripts/test_package_and_sign.sh
./scripts/test_package_and_sign.sh

Logging Levels¶

Control log verbosity: 

# Production (INFO)
RUST_LOG=info cargo run --bin service-http

# Development (DEBUG)
RUST_LOG=debug cargo run --bin service-http

# Full Debug (TRACE)
RUST_LOG=trace cargo run --bin service-http
