Service-HTTP Error Handling Refactor¶

Overview¶

Refactored the service-http crate to use thiserror for structured, type-safe error handling that provides predictable error messages without leaking internal implementation details.

Date: March 9, 2026 Scope: crates/service-http/src/v2/

Goals Achieved¶

  1. ✅ Type-safe error handling using thiserror derive macros
  2. ✅ Separation of concerns - internal errors tracked separately from user-facing messages
  3. ✅ No information leakage - sensitive details logged but not exposed to API clients
  4. ✅ Automatic error logging - internal errors logged with full context
  5. ✅ Reduced boilerplate - leveraging From trait implementations
  6. ✅ Consistent error responses - uniform JSON structure across all endpoints

Architecture¶

Error Type Hierarchy¶

pub enum ApiError {
    InvalidInput { message, source },          // 400 - Client provided invalid input
    InvalidInputType { message, input_index, source }, // 400 - Wrong input type for processor
    ValidationFailed { message, source },      // 400 - Schema/constraint validation failed
    JobNotFound { job_id },                    // 404 - Job ID doesn't exist
    Timeout,                                   // 408 - Sync request exceeded timeout
    ProcessingFailed { source },               // 500 - Job submission/processing error
    Internal { source },                       // 500 - Unexpected internal error
}

Key Design Principles¶

1. Source Tracking with #[source]¶

Internal errors use #[source] attribute to maintain error chains for debugging:

#[error("Processing failed")]
ProcessingFailed {
    #[source]
    source: anyhow::Error,  // Full error chain preserved
}

2. Safe User Messages¶

The user_message() method provides sanitized, generic messages for API responses:

fn user_message(&self) -> String {
    match self {
        Self::Internal { .. } => {
            "An internal error occurred. Please try again later.".to_string()
        }
        Self::ProcessingFailed { .. } => {
            "Processing failed. Please check your request and try again.".to_string()
        }
        // ... specific messages for client errors
    }
}

3. Automatic Error Logging¶

Internal errors are automatically logged with full context in IntoResponse:

match &self {
    Self::Internal { source } => {
        error!(
            error = %source,
            error_chain = ?source,
            "Internal server error"
        );
    }
    // ...
}

4. Automatic Conversions via From¶

Common error types auto-convert to ApiError:

impl From<anyhow::Error> for ApiError
impl From<garde::Report> for ApiError
impl From<serde_json::Error> for ApiError

This enables clean handler code: 

// Before
req.validate()
    .map_err(|e| ApiError::validation_failed(e.to_string()))?;

// After
req.validate()?;  // Automatic conversion via From<garde::Report>

API Response Format¶

All errors return a consistent JSON structure:

{
  "error": {
    "code": "VALIDATION_FAILED",
    "message": "Validation failed: inputs must not be empty",
    "details": {
      "input_index": 0,
      "field": "path"
    }
  }
}

Error Codes¶

Code HTTP Status Description
INVALID_INPUT 400 Client provided invalid input format
INVALID_INPUT_TYPE 400 Input type doesn't match processor requirements
VALIDATION_FAILED 400 Request failed schema/constraint validation
JOB_NOT_FOUND 404 Job ID not found in store
REQUEST_TIMEOUT 408 Sync request exceeded configured timeout
PROCESSING_FAILED 500 Job submission or processing error
INTERNAL_ERROR 500 Unexpected internal error

Handler Patterns¶

Before Refactor¶

pub async fn c2pa_video_async(
    State(state): State<AppState>,
    Json(req): Json<C2paVideoRequest>,
) -> Result<Json<JobResponse>, ApiError> {
    // Verbose error mapping
    req.validate()
        .map_err(|e| ApiError::validation_failed(e.to_string()))?;

    VideoInputValidator.validate_inputs(&req.inputs)?;

    let spec = req
        .try_into()
        .map_err(|e: serde_json::Error| ApiError::internal_error(e.to_string()))?;

    submit_job_async(&state, spec).await
}

After Refactor¶

pub async fn c2pa_video_async(
    State(state): State<AppState>,
    Json(req): Json<C2paVideoRequest>,
) -> Result<Json<JobResponse>, ApiError> {
    // Clean, automatic error conversions
    req.validate()?;
    VideoInputValidator.validate_inputs(&req.inputs)?;
    let spec = req.try_into()?;
    submit_job_async(&state, spec).await
}

Security Improvements¶

Information Leakage Prevention¶

Before: Error messages exposed internal details 

ApiError::internal_error(format!("Database connection failed: {}", db_error))
// Response: "Database connection failed: Connection refused at localhost:5432"

After: Generic messages, details only in logs 

ApiError::internal(db_error.into())
// Response: "An internal error occurred. Please try again later."
// Logs: Full error chain with connection details

Logged vs. Exposed¶

Error Type User Message Logged Details
Internal "An internal error occurred" Full error chain, source file, line number
ProcessingFailed "Processing failed. Check your request" Anyhow error with context chain
ValidationFailed Specific validation message Validation report (non-sensitive)
JobNotFound "Job not found: {id}" Job ID only

Best Practices Applied¶

1. Error Type Hierarchy¶

  • Use enum variants for different error categories
  • thiserror derives Error trait implementation
  • Each variant maps to appropriate HTTP status

2. Source Error Tracking¶

  • #[source] attribute preserves error chains
  • Full context available for debugging
  • Supports error downcasting if needed

3. Display vs. Debug¶

  • Display shows user-safe messages
  • Debug includes full error details for logging
  • IntoResponse uses Display for API responses

4. Automatic Conversions¶

  • From implementations reduce boilerplate
  • Handlers use ? operator cleanly
  • Type inference handles conversions

5. Contextual Error Creation¶

  • Constructor methods provide semantic clarity
  • validation_failed(), processing_failed(), etc.
  • Source errors attached where relevant

Testing Recommendations¶

Unit Tests¶

#[test]
fn test_error_messages_dont_leak_internals() {
    let internal_err = anyhow::anyhow!("Database password: secret123");
    let api_err = ApiError::internal(internal_err);

    let message = api_err.user_message();
    assert!(!message.contains("secret123"));
    assert_eq!(message, "An internal error occurred. Please try again later.");
}

Integration Tests¶

#[tokio::test]
async fn test_invalid_input_returns_400() {
    let response = client
        .post("/v2/c2pa/video")
        .json(&invalid_request)
        .send()
        .await?;

    assert_eq!(response.status(), 400);

    let error: ApiErrorResponse = response.json().await?;
    assert_eq!(error.error.code, "VALIDATION_FAILED");
    assert!(!error.error.message.contains("internal"));
}

Migration Notes¶

For New Endpoints¶

  1. Use ApiError as handler return type
  2. Leverage ? operator for error propagation
  3. Use semantic constructors: ApiError::validation_failed(), etc.
  4. Let From implementations handle conversions

For Existing Code¶

  1. Replace .map_err(|e| ApiError::internal_error(e.to_string())) with .map_err(ApiError::internal)
  2. Replace garde validation mapping with direct ?
  3. Remove explicit error string formatting for internal errors
