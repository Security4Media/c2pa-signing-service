App Crate Error Handling Refactor¶

Overview¶

Refactored the app crate to use thiserror for structured, type-safe error handling that provides clear error boundaries between application orchestration layer and domain/infrastructure layers.

Date: March 9, 2026
Scope: crates/app/
Architecture: Hexagonal (Ports & Adapters)

Goals Achieved¶

  1. ✅ Type-safe error handling using thiserror derive macros
  2. ✅ Clear error boundaries - distinct error types for different concerns
  3. ✅ Rich context tracking - full error chains via #[source] attribute
  4. ✅ Hexagonal architecture compliance - errors respect layer boundaries
  5. ✅ Automatic error propagation - leveraging ? operator and From traits
  6. ✅ Improved debugging - structured error context for logs and traces

Architecture¶

Error Type Hierarchy¶

pub enum AppError {
    JobSubmission { source: JobError },      // Job submission failed
    JobStatusQuery { source: JobError },     // Status retrieval failed
    JobProcessing { source: JobError },      // Job processing failed
    Registry { source: RegistryError },      // Processor registry error
    Configuration { message, source },       // Configuration error
    JobStore { source: anyhow::Error },      // Job store operation failed
    Channel { source: Box<dyn Error> },      // Channel communication failed
}

pub enum RegistryError {
    ProcessorNotFound { kind: ProcessorKind },    // Processor not registered
    ConfigurationFile { path, source },           // Config file error
    ProcessorInitialization { kind, source },     // Processor init failed
}

pub enum JobError {
    ValidationFailed { source },        // Batch spec validation failed
    NotFound { job_id },                // Job not found in store
    StoreFailed { source },             // Job store operation failed
    InputResolution { source },         // Input resolution failed
    ProcessorExecution { source },      // Processor execution failed
    OutputSink { source },              // Output sink operation failed
    SemaphoreAcquisition,               // Internal sync error
    EnqueueFailed { source },           // Channel enqueue failed
    UnsupportedArtifact { artifact_type }, // Unsupported output type
}

Key Design Principles¶

1. Separation of Concerns¶

Each error type addresses a specific concern: - JobError: Job lifecycle operations (validation, execution, storage) - RegistryError: Processor discovery and initialization - AppError: Top-level application orchestration

2. Source Error Tracking¶

All errors preserve source context via #[source]:

#[derive(Error, Debug)]
pub enum JobError {
    #[error("Processor execution failed")]
    ProcessorExecution {
        #[source]
        source: anyhow::Error,  // Full error chain preserved
    },
    // ...
}

3. Semantic Constructors¶

Helper methods provide semantic clarity:

impl JobError {
    pub fn processor_execution(err: anyhow::Error) -> Self {
        JobError::ProcessorExecution { source: err }
    }

    pub fn not_found(job_id: impl Into<String>) -> Self {
        JobError::NotFound { job_id: job_id.into() }
    }
}

4. Layer-Appropriate Error Handling¶

Each layer handles only its concerns:

// Runner (app layer) - converts infrastructure errors to JobError
let artifact = processor
    .process_boxed(input, ctx, resolvers)
    .await
    .map_err(JobError::processor_execution)?;

// BatchService - converts JobError to AppError with context
let id = self.job_store
    .create_job(spec)
    .await
    .map_err(|e| AppError::JobSubmission {
        source: JobError::store(e),
    })?;

Updated Components¶

1. batch_service.rs¶

Before: 

async fn submit(&self, spec: BatchSpec) -> anyhow::Result<BatchId> {
    validate_spec(&spec)?;
    let id = self.job_store.create_job(spec).await?;
    self.sender.send(id.clone()).await?;
    Ok(id)
}

After: 

async fn submit(&self, spec: BatchSpec) -> Result<BatchId, AppError> {
    validate_spec(&spec).map_err(|e| AppError::JobSubmission {
        source: JobError::validation(e),
    })?;

    let id = self.job_store
        .create_job(spec)
        .await
        .map_err(|e| AppError::JobSubmission {
            source: JobError::store(e),
        })?;

    self.sender
        .send(id.clone())
        .await
        .map_err(|e| AppError::JobSubmission {
            source: JobError::EnqueueFailed { source: Box::new(e) },
        })?;

    Ok(id)
}

Benefits: - Clear error categorization (validation vs store vs channel) - Rich context for debugging - Type-safe error handling

async fn handle_job(
    batch_id: BatchId,
    deps: WorkerDeps,
    config: WorkerConfig,

    let snapshot = deps.job_store
        .get_job(&batch_id)
        .await
        .map_err(JobError::store)?;

    let processor = deps.processor_registry
        .get(&snapshot.spec.processor)
        .map_err(|e| JobError::ProcessorExecution {
            source: anyhow::anyhow!("Processor not found: {}", e),
        })?;
    // ...
}

Benefits: - Explicit error type (JobError instead of generic anyhow) - Structured logging with error source tracking - Better error categorization for metrics

3. registry/mod.rs & map_registry.rs¶

Before:

    fn get(&self, spec: &ProcessorSpec) -> anyhow::Result<Arc<dyn DynProcessor>>;
}

impl ProcessorRegistry for MapProcessorRegistry {
    fn get(&self, spec: &ProcessorSpec) -> Result<Arc<dyn DynProcessor>, anyhow::Error> {
        self.map
            .get(&spec.kind)
            .cloned()
            .ok_or_else(|| anyhow::anyhow!("Unknown processor kind: {:?}", spec.kind))
    }
}

After: 

pub trait ProcessorRegistry: Send + Sync {
    fn get(&self, spec: &ProcessorSpec) -> Result<Arc<dyn DynProcessor>, RegistryError>;
}

impl ProcessorRegistry for MapProcessorRegistry {
    fn get(&self, spec: &ProcessorSpec) -> Result<Arc<dyn DynProcessor>, RegistryError> {
        self.map
            .get(&spec.kind)
            .cloned()
            .ok_or_else(|| RegistryError::processor_not_found(spec.kind))
    }
}

Benefits: - Type-safe error with ProcessorKind context - Clear error message: "Unknown processor kind: C2paSign" - Easier to handle specific error cases

     }
   }

8. Logs (with full context):
   ERROR processor registry error: Unknown processor kind: C2paVideo
         error_chain=RegistryError::ProcessorNotFound

Integration with service-http¶

The service-http layer automatically converts AppError to ApiError:

impl From<app::AppError> for ApiError {
    fn from(err: app::AppError) -> Self {
        use app::AppError;
        match err {
            AppError::JobSubmission { source } | AppError::JobProcessing { source } => {
                match source {
                    JobError::ValidationFailed { .. } => 
                        Self::validation_failed(...),
                    JobError::NotFound { job_id } => 
                        Self::job_not_found(job_id),
                    JobError::StoreFailed { source } => 
                        Self::processing_failed(source),
                    // ... map all variants
                }
            }
            AppError::JobStatusQuery { source } => { /* ... */ }
            AppError::Registry { source } => 
                Self::internal(anyhow::anyhow!("Processor registry error: {}", source)),
            // ... handle all variants
        }
    }
}

This enables clean handler code:

pub async fn submit_job_async(
    state: &AppState,
    spec: BatchSpec,
) -> Result<Json<JobResponse>, ApiError> {
    let batch_id = state.service.submit(spec).await?;  // Automatic conversion!
    Ok(Json(JobResponse { /* ... */ }))
}

Best Practices Applied¶

1. Error Granularity¶

Each error variant represents a distinct failure mode: - JobError::ValidationFailed vs JobError::ProcessorExecution - RegistryError::ProcessorNotFound vs RegistryError::ConfigurationFile

2. Context Preservation¶

All errors preserve source context: 

#[error("Processor execution failed")]
ProcessorExecution {
    #[source]
    source: anyhow::Error,  // Full chain available
}

3. Semantic Constructors¶

Helper methods improve readability: 

// Instead of:
JobError::StoreFailed { source: err }

// Use:
JobError::store(err)

4. Display Messages¶

Error messages are human-readable and actionable: 

#[error("Job not found: {job_id}")]
NotFound { job_id: String }

// Produces: "Job not found: batch-abc-123"

5. Logging Integration¶

Errors include rich context for logging: 

error!(
    error = %err,
    error_source = ?err.source(),
    "batch processing failed"
);
