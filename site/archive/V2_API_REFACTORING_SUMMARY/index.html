API v2 Refactoring - Implementation Summary¶

Date: March 4, 2026
Status: ✅ Complete and Tested

Overview¶

Successfully refactored the HTTP API from a batch-centric design to a processor-centric design with feature-gated versioning. The new v2 API provides cleaner endpoints, better validation, and improved developer experience while maintaining backwards compatibility through feature flags.

Key Changes¶

1. Feature-Gated API Versions¶

  • Added v1_api and v2_api feature flags to service-http/Cargo.toml
  • Default feature is v2_api
  • Compilation enforces at least one API version is enabled
  • Clean separation allows independent evolution of API versions

2. New URL Structure (v2)¶

Old (v1): - POST /v1/batches - Submit batch job - POST /v1/batches/sync - Submit and wait - GET /v1/batches/{id} - Get batch status

New (v2): - POST /v2/c2pa/video - Sign video files - POST /v2/c2pa/fragmented - Sign HLS content - POST /v2/package - Package media - POST /v2/c2pa/package - Package and sign - POST /v2/sync/... - Sync versions of all endpoints - GET /v2/jobs/{id} - Get job status - GET /v2/health - Health check

3. Simplified Request Format¶

Old (v1): 

{
  "inputs": [...],
  "processor": {
    "kind": "c2pa_sign",
    "params": {...}
  },
  "output": {...}
}

New (v2): 

{
  "inputs": [...],
  "params": {...},
  "output": {...},
  "options": {...}
}

Processor type is implicit from the endpoint URL, not specified in the request body.

4. Enhanced Response Format¶

v2 Response includes: - job_id - Unique identifier - status - "queued", "running", "succeeded", "failed" - results[] - Per-input results with status and error messages - timing - ISO 8601 timestamps (queued_at, started_at, finished_at)

5. Input Validation Per Endpoint¶

Each endpoint validates that inputs match processor requirements:

  • /c2pa/video - Accepts file inputs only (rejects folders)
  • /c2pa/fragmented - Accepts HLS publication or folder inputs
  • /package - Accepts folder inputs containing MP4 files
  • /c2pa/package - Accepts folder inputs for packaging + signing

Architecture¶

Module Structure¶

service-http/src/
├── main.rs                    # Feature-gated router initialization
├── middleware.rs              # Shared middleware
├── multi_form/                # Multipart form handling
├── processing.rs              # Processor registry builder
├── v1/                        # V1 API (old batch-based)
│   ├── mod.rs
│   ├── dto.rs                 # V1 DTOs
│   ├── handlers.rs            # V1 handlers
│   └── routes.rs              # V1 routes
└── v2/                        # V2 API (processor-centric)
    ├── mod.rs
    ├── conversions.rs         # Domain → DTO conversions
    ├── dto.rs                 # V2 request/response DTOs
    ├── error.rs               # Structured API errors
    ├── handlers.rs            # V2 endpoint handlers
    ├── routes.rs              # V2 routes
    └── validation.rs          # Input validators

Key Design Decisions¶

1. Reused Existing JobStore¶

  • Did not create duplicate job storage
  • Uses domain::JobStore trait already implemented in infra::InMemoryJobStore
  • Extensible to persistent storage (database) in the future

2. Domain Type Conversions¶

  • Created conversions.rs module to map BatchSnapshot to v2 DTOs
  • Handles timestamp conversion (ms → ISO 8601)
  • Maps FileResult fields correctly (message → error, output.location → output_path)

3. Input Validation¶

  • Created validator traits for each processor type
  • Compile-time safety through Rust's type system
  • Runtime validation with helpful error messages

4. Error Handling¶

  • Unified ApiError type with error codes
  • Standardized error response format
  • Includes details for debugging (input_index, field, etc.)

Implementation Details¶

DTOs Created¶

Request DTOs: - C2paVideoRequest / C2paVideoParams - C2paFragmentedRequest / C2paFragmentedParams - PackageRequest / PackageParams - PackageAndSignRequest / PackageAndSignParams - ProcessingOptions (shared) - OutputSpec (shared)

Response DTOs: - JobResponse - Async submission response - JobResultResponse - Job completion result - ProcessingResult - Per-input result - JobTiming - Timestamp information

Handlers Implemented¶

Each processor has two handlers: 1. *_async - Returns job_id immediately 2. *_sync - Blocks until completion or timeout

Implemented handlers: - c2pa_video_async / c2pa_video_sync - c2pa_fragmented_async / c2pa_fragmented_sync - package_async / package_sync - package_and_sign_async / package_and_sign_sync - get_job_status - health

Validation¶

Created three validators: - VideoInputValidator - Validates video file inputs - FragmentedInputValidator - Validates HLS/folder inputs - PackageInputValidator - Validates folder inputs

Configuration¶

Environment Variables¶

  • SYNC_TIMEOUT_MS - Timeout for sync endpoints (default: 120000 ms)
  • SYNC_POLL_MS - Polling interval (default: 250 ms)
  • RUST_LOG - Log level configuration

Build Features¶

# V1 API only
cargo build --features v1_api

# V2 API only (default)
cargo build --features v2_api

# Both APIs (not supported - mutually exclusive)
# Will result in compile error if both are enabled

Migration Guide (v1 → v2)¶

For API Consumers¶

Old v1 request: 

curl -X POST http://localhost:8080/v1/batches \
  -H "Content-Type: application/json" \
  -d '{
    "inputs": [{"type": "local_file_path", "path": "/video.mp4"}],
    "processor": {"kind": "c2pa_sign", "params": {}},
    "output": {"type": "local_path", "base": "/output"}
  }'

New v2 request: 

curl -X POST http://localhost:8080/v2/c2pa/video \
  -H "Content-Type: application/json" \
  -d '{
    "inputs": [{"type": "local_file_path", "path": "/video.mp4"}],
    "params": {},
    "output": {"type": "local_path", "base": "/output"}
  }'

Key differences: 1. URL changed from /v1/batches to /v2/c2pa/video 2. Removed processor wrapper - processor type is in URL 3. params is top-level instead of nested 4. Added /v2/sync/... variants for synchronous execution

Lessons Learned¶

  1. Domain Reuse: Leveraging existing JobStore trait saved time and ensured consistency
  2. Feature Flags: Rust's feature system provides clean API versioning
  3. Type Safety: Input validation at the handler level prevents processor misuse
  4. Conversions: Separate conversion layer keeps domain and API concerns separate

Conclusion¶

The v2 API refactoring successfully achieves:

✅ Processor-centric URL structure
✅ Simplified request format
✅ Enhanced response format with detailed results
✅ Input validation per endpoint
✅ Sync and async operation modes
✅ Feature-gated versioning
✅ Backwards compatibility through v1 API
✅ Extensible architecture
✅ Clean separation of concerns

The implementation is production-ready and provides a solid foundation for future enhancements.
