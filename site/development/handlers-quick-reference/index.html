Handler Quick Reference¶

Current Layout¶

crates/service-http/src/v2/handlers/ is organized by public API capability:

  • status.rs for GET /v2/health and GET /v2/jobs/{id}
  • c2pa_video.rs for monolithic video signing
  • c2pa_fragmented.rs for fragmented/HLS signing
  • package.rs for packaging only
  • c2pa_package.rs for package-and-sign flows
  • softbinding.rs for placeholder watermark endpoints
  • common.rs for shared async/sync submission helpers

Request Handling Pattern¶

Each processing handler follows the same shape:

  1. Deserialize a request DTO from crate::v2::dto::*
  2. Validate the DTO with garde
  3. Validate processor/input compatibility with crate::v2::validation::*InputValidator
  4. Convert the DTO into domain::BatchSpec
  5. Call submit_job_async or submit_job_sync

Response Conventions¶

  • Async processing endpoints return Json<JobResponse> with job_id and status
  • Sync processing endpoints return (StatusCode::OK, Json<JobResultResponse>)
  • Timeouts are surfaced as ApiError::timeout() and become HTTP 408
  • Placeholder watermark endpoints return HTTP 501 with body Coming soon

State And Shared Constraints¶

Handlers receive AppState from crates/service-http/src/v2/routes.rs:

  • service: Arc<dyn BatchService>
  • sync_timeout_ms
  • sync_poll_interval_ms
  • allowed_s3_output_buckets

submit_job_async and submit_job_sync enforce:

  • BatchSpec::validate_spec()
  • S3 output validation against the configured allow-list

Adding A New Processing Endpoint¶

  1. Add the request DTO under crates/service-http/src/v2/dto/
  2. Add or reuse an input validator in crates/service-http/src/v2/validation.rs
  3. Implement async and sync handlers under crates/service-http/src/v2/handlers/
  4. Register both routes in crates/service-http/src/v2/routes.rs
  5. If the endpoint is public, add utoipa::path annotations so the OpenAPI doc stays complete
  6. Extend crates/service-http/src/v2/docs.rs tests if the public route set changes

Testing¶

  • Handler unit tests live beside the handlers
  • Router coverage lives in crates/service-http/src/v2/routes.rs
  • Integration behavior is exercised in worker_integration.rs
