Testing Strategy¶

Overview¶

The c2pa-claim-generator project follows a comprehensive testing strategy with multiple levels of testing to ensure reliability, correctness, and performance.

Test Levels¶

1. Unit Tests¶

Location: Within each crate using #[cfg(test)] modules

Purpose: Test individual functions, methods, and components in isolation

Coverage: - Domain logic validation - Processor implementations - Input resolution logic - Configuration parsing - Error handling paths - Remote URL dispatch and S3 URL parsing

Example: 

#[cfg(test)]
mod tests {
    use super::*;

    #[test]
    fn test_batch_spec_validation() {
        // Test validation logic
    }
}

Running: 

cargo test --lib

2. Integration Tests¶

Location: tests/integration/

Purpose: Test interactions between crates and components

Coverage: - Cross-crate interactions - Processor registry - Job lifecycle management - Input resolver composition - Output sink operations - S3 startup validation and object staging when the s3 feature is enabled

Structure: 

tests/integration/
├── c2pa_signing_test.rs        # C2PA signing integration
├── media_packaging_test.rs     # Media packaging integration
└── api_endpoints_test.rs       # API endpoint integration

Running: 

cargo test --test integration_test_name

3. End-to-End Tests¶

Location: tests/e2e/

Purpose: Test complete workflows from HTTP request to output

Coverage: - Full HTTP API workflows - Async and sync endpoints - Real media file processing - Error scenarios - Concurrent job processing

Example workflow: 1. Start server 2. Submit job via HTTP 3. Poll for completion 4. Verify output files 5. Validate C2PA claims

Running: 

cargo test --test e2e_test_name

4. Benchmarks¶

Location: benches/

Purpose: Performance testing and regression detection

Coverage: - C2PA signing performance - Media packaging throughput - Concurrent job processing - HTTP API latency

Structure: 

benches/
├── c2pa_signing_bench.rs       # C2PA signing benchmarks
├── media_packaging_bench.rs    # Packaging benchmarks
├── concurrent_jobs_bench.rs    # Concurrency benchmarks
└── api_throughput_bench.rs     # API performance benchmarks

Running: 

cargo bench

Fixture Guidelines¶

  1. Minimal Size: Keep test files as small as possible
  2. Representative: Cover common formats and edge cases
  3. Documented: Include README in each fixture directory explaining files
  4. Reusable: Shared across unit, integration, and e2e tests
  5. Version Controlled: Check into Git (use LFS if needed)

Testing Best Practices¶

CI Validation Gate¶

The default CI gate is intentionally narrow and should stay fast and stable:

cargo fmt --all --check
cargo clippy-default
cargo test --workspace --no-run

That gate is the required baseline for normal pull requests.

Follow-up Validation Backlog¶

Add these only after the default gate is stable:

  • cargo clippy-all
  • selected feature-matrix checks, especially infra resolver combinations and service-http --features s3
  • service-http docs feature checks:
  • cargo check -p service-http --features openapi
  • cargo check -p service-http --features "openapi scalar-docs"
  • cargo test -p service-http --features openapi
  • cargo test -p service-http --features "openapi scalar-docs"
  • full cargo test --workspace
  • stable line/region/function coverage jobs

Do not add branch coverage to CI until the LLVM crash in the branch reporting path is resolved.

1. Test Isolation¶

  • Each test should be independent
  • Clean up artifacts after tests
  • Use temporary directories for output
  • Mock external dependencies

2. Test Naming¶

#[test]
fn test_<component>_<scenario>_<expected_outcome>() {
    // Example: test_c2pa_processor_invalid_input_returns_error
}

3. Assertions¶

  • Use descriptive assertion messages
  • Test both success and failure paths
  • Validate side effects (files created, logs written)
  • Prefer assert_matches! for enum/result/option variants when the type is the contract under test
  • Assert on error variants/types rather than error strings
  • Only assert on error messages when the message itself is a user-facing or API-facing contract
use std::assert_matches::assert_matches;

assert_matches!(result, Ok(_));
assert_matches!(err, DomainError::InputResolution);

3a. Test Refactor Step¶

As part of ongoing test maintenance, add a refactor pass that updates older tests to follow the same pattern:

  • replace string-based error assertions with type/variant assertions where the error type is the real contract
  • replace assert!(matches!(...)) with assert_matches! unless plain boolean assertions are materially clearer

4. Async Tests¶

#[tokio::test]
async fn test_async_operation() {
    let result = async_function().await;
    assert!(result.is_ok());
}

5. Parallel Execution¶

  • Tests run in parallel by default
  • Use #[serial] attribute for tests requiring serialization
  • Avoid shared mutable state

6. Test Coverage¶

# Install coverage tool
cargo install cargo-llvm-cov

# Summary report
cargo llvm-cov --workspace --summary-only

# HTML report
cargo llvm-cov --workspace --html

# Summary report with branch coverage
./scripts/dev/cov-branches.sh

Branch coverage is currently an unstable cargo-llvm-cov feature. Use the repo script instead of raw cargo llvm-cov --branch so coverage export uses the LLVM tools bundled with the active Rust toolchain rather than the system LLVM.

CI/CD Integration¶

GitHub Actions Example¶

name: Tests

on: [push, pull_request]

jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v3

      - name: Install dependencies
        run: |
          sudo apt-get update
          sudo apt-get install -y ffmpeg
          # Install Bento4

      - name: Run tests
        run: cargo test --workspace

      - name: Run benchmarks
        run: cargo bench --no-run  # Just compile

Performance Testing¶

Load Testing¶

Location: scripts/load/

Tools: Python + locust or similar

Scenarios: - Single file processing - Batch processing (10, 50, 100 inputs) - Concurrent job submission - Long-running jobs

Running: 

cd scripts/load
pip install -r requirements.txt
python load_test.py

Metrics to Track¶

  • Throughput: Jobs processed per second
  • Latency: Time from submission to completion
  • Resource Usage: CPU, memory, disk I/O
  • Error Rate: Failed jobs per total jobs

rust-gdb target/debug/deps/test_binary

Continuous Improvement¶

Code Coverage Goals¶

  • Domain crate: 90%+ coverage (core logic)
  • App crate: 80%+ coverage (orchestration)
  • Infra crate: 70%+ coverage (implementations)
  • Service-http crate: 80%+ coverage (API handlers)

Coverage Rollout¶

  • Start with report-only coverage in CI
  • Use crate-by-crate coverage review to drive missing test cases
  • Add thresholds only after the unit-test baseline is in place

Mutation Testing¶

Consider using cargo-mutants to test the quality of tests:

cargo install cargo-mutants
cargo mutants
