Architecture Overview¶

System Design¶

The c2pa-claim-generator is a high-performance, async batch processing service for C2PA signing and media packaging operations. It follows a hexagonal architecture (ports and adapters) pattern with clear separation of concerns.

Core Principles¶

  1. Dependency Inversion: Domain defines contracts, infrastructure provides implementations
  2. Type Safety: Compile-time guarantees through Rust's type system
  3. Async-First: Built on Tokio for concurrent processing
  4. Extensibility: Trait-based abstractions allow easy addition of new processors
  5. Configuration-Driven: Runtime behavior controlled through configuration files

Crates¶

domain¶

Responsibility: Core business logic and abstractions

  • Defines all traits (Processor, InputResolver, OutputSink, JobStore)
  • Domain models (BatchSpec, InputRef, ProcessedArtifact, etc.)
  • No external dependencies except serde and async-trait
  • Stability: This is the stable 1.x+ API surface

app¶

Responsibility: Application orchestration

  • BatchService implementation
  • Worker pool management
  • Job queue and lifecycle management
  • ProcessorRegistry
  • Coordinates domain and infra layers

infra¶

Responsibility: Concrete implementations

  • Processors: C2PA signing (monolithic and fragmented), media packaging, combined operations
  • Resolvers: Local filesystem, HTTP downloads, S3
  • Stores: In-memory job storage, (future: SQLite/PostgreSQL)
  • Sinks: Local filesystem output, (future: S3, GCS)

Feature flags: - local - Local filesystem support - http - HTTP input resolver - s3 - AWS S3 input resolver support

service-http¶

Responsibility: HTTP API layer

  • Axum-based REST API
  • Request/response DTOs
  • Validation using garde
  • Endpoint handlers
  • Sync and async endpoint modes

Processing Pipeline¶

1. Request Ingestion¶

HTTP Request → Validation → DTO Conversion → BatchSpec

2. Job Creation¶

BatchSpec → BatchService → JobStore → Job ID

3. Worker Processing¶

Worker Pool → Dequeue Job → Resolve Inputs → Process → Write Output

4. Resolution Phase¶

InputRef → InputResolver → ResolvedInput (typed per processor)

5. Processing Phase¶

ResolvedInput → Processor<Input> → ProcessedArtifact

6. Output Phase¶

ProcessedArtifact → OutputSink → OutputRef

Processor System¶

Type-Safe Processing¶

Each processor declares its input type:

trait Processor {
    type Input;
    async fn process(&self, input: Self::Input, ctx: ProcessContext) 
        -> Result<ProcessedArtifact>;
}

Examples: - C2paMonolithicProcessor: Processor<Input = ResolvedFileInput> - C2paFmp4Processor: Processor<Input = ResolvedHlsPublication> - MediaPackagerProcessor: Processor<Input = ResolvedFolderInput>

Processor Registry¶

Dynamic dispatch through DynProcessor trait objects stored in a registry keyed by processor kind.

Concurrency Model¶

Worker Pool¶

  • Configurable number of workers (default: CPU cores)
  • Each worker runs an infinite loop processing jobs from a queue
  • Tokio-based async concurrency

Per-Job Parallelism¶

  • Each job can process multiple inputs in parallel
  • Controlled by max_files_parallel option (default: 4)
  • Futures spawned with semaphore-based rate limiting

Sync vs Async Endpoints¶

  • Async endpoints: Return job ID immediately, client polls
  • Sync endpoints: Block and poll internally until completion or timeout

Error Handling¶

Error Hierarchy¶

ProcessorError (domain)
    ├── InputResolutionError
    ├── ProcessingError
    └── OutputSinkError

MediaPackagerError (media-packager)
    ├── CommandNotFound
    ├── CommandFailed
    └── InvalidConfig

C2paError (from c2pa crate)

Error Propagation¶

  • Errors wrapped in anyhow::Error for flexibility
  • Per-input error tracking in job results
  • Fail-fast for fatal errors, continue for input-specific failures

Configuration¶

Runtime Configuration Files¶

Located in config/: - service.toml: Top-level server, worker, resolver, sink, and processor path settings - c2pa/: C2PA SDK settings, signer configurations - media-packager/: Pipeline configurations, representation metadata - signers/: Remote signer endpoints and credentials - keys/: Public keys and certificates

Environment Variables¶

  • RUST_LOG - Logging level (info, debug, trace)
  • SERVICE_CONFIG_PATH - Alternate top-level service config file
  • SERVICE__... - Overrides for values in config/service.toml

Testing Strategy¶

Unit Tests¶

  • Within each crate using #[cfg(test)] modules
  • Mock implementations of traits for isolated testing

Integration Tests¶

  • Located in tests/integration/
  • Test cross-crate interactions
  • Use in-memory stores and local filesystem

End-to-End Tests¶

  • Located in tests/e2e/
  • Full pipeline tests with real media files
  • HTTP API tests

Test Fixtures¶

  • Located in tests/fixtures/
  • Organized by media type (video, audio, image, HLS, DASH)
  • Shared across all test types

Benchmarks¶

  • Located in benches/
  • Performance testing for critical paths
  • Concurrent job processing benchmarks

Security Considerations¶

C2PA Signing¶

  • Key storage (filesystem, clound HSM)
  • Remote signing support
  • Manifest validation

Input Validation¶

  • Schema validation using garde
  • Path traversal prevention
  • File type verification

Output Control¶

  • Sandboxed output directories
  • Configurable base paths
  • Prevent overwrites (by default)

Deployment¶

Docker¶

  • Multi-stage build for optimized images
  • FFmpeg and Bento4 compiled from source
  • Runtime stage includes only necessary binaries
  • See Dockerfile for details

Dependencies¶

  • Runtime: FFmpeg 4.4+, Bento4 1.6.0+
  • Build: Rust 2024 edition, cargo

Performance Characteristics¶

Throughput¶

  • Scales with number of workers (CPU cores)
  • I/O bound for large video files
  • CPU bound for C2PA signing

Latency¶

  • Async mode: <50ms to job submission
  • Sync mode: Depends on processing time + polling overhead

Resource Usage¶

  • Memory: ~100MB base + per-job allocations
  • CPU: Scales with parallelism settings
  • Disk: Temporary files cleaned up automatically

Future Enhancements¶

  • Image C2PA signing
  • S3/GCS input resolvers and output sinks
  • Authentication and authorization
  • Persistent job storage (SQLite/PostgreSQL)
  • Live DASH streaming support
  • Webhook notifications for job completion
