Crate Organization¶

Overview¶

This document describes the organization and architecture of the c2pa-claim-generator workspace.

Workspace Structure¶

c2pa-claim-generator/
├── crates/
    ├── domain/          # Core business logic & traits (no external deps)
    ├── app/             # Application orchestration layer
    ├── infra/           # Infrastructure implementations
    └── service-http/    # HTTP API layer (Axum)

Dependency Graph¶

service-http
    ↓
  app  ←──────┐
    ↓         │
  infra ──────┤
    ↓         │
 domain ←─────┘

media-packager (standalone, external repository, used by infra)

Architecture Principles¶

1. Clean Dependency Inversion (Ports & Adapters)¶

  • domain defines traits (Processor, JobStore, InputResolvers, OutputSink)
  • infra provides concrete implementations
  • app orchestrates without knowing implementation details
  • Follows hexagonal architecture principles

2. Type-Safe Processor System¶

  • Generic Processor<Input> trait with associated types
  • Type erasure via DynProcessor for dynamic dispatch
  • Clean separation: processors in infra, registry in app

3. Feature Flags for Resolvers¶

  • infra exposes local, http, s3, etc. features
  • service-http selects only what it needs
  • Reduces binary size and dependencies

Crate Responsibilities¶

domain¶

Purpose: Core business logic, traits, and domain models

Contains: - Processor trait and base types - Job management abstractions (JobStore, BatchService) - Input/output abstractions (InputResolver, OutputSink) - Domain models (BatchSpec, ProcessedArtifact, etc.)

Dependencies: Minimal—only serialization and async traits

app¶

Purpose: Application orchestration and runtime

Contains: - BatchService implementation - Worker pool management - Processor registry - Job execution logic

Dependencies: domain, infra

infra¶

Purpose: Concrete implementations of domain ports

Contains: - Processors: C2PA signing, media packaging - Input resolvers: local filesystem, HTTP, S3 - Output sinks: local filesystem - Job stores: in-memory, (future: SQLite)

Dependencies: domain, external libraries (c2pa, media-packager, reqwest, etc.)

Features: - local: Local filesystem support - http: HTTP input resolver - s3: AWS S3 input resolver support

Remote object-storage inputs stay provider-agnostic in domain. The HTTP API accepts remote_file_url and remote_folder_url, and infra dispatches by URL scheme to the appropriate resolver implementation.

service-http¶

Purpose: HTTP API layer using Axum

Contains: - REST API endpoints - Request/response DTOs - HTTP-specific middleware - Server configuration

Dependencies: app, domain, infra, axum

Public API Design¶

Stability Guarantees¶

Stable API (1.x+): - domain::Processor trait - domain::BatchService - domain::JobStore, domain::InputResolver, domain::OutputSink - Core models: BatchSpec, InputRef, OutputRef, ProcessedArtifact

Experimental (may change in minor versions): - Resolved input types (ResolvedMediaInput, etc.) - Media-specific processors

Organization Principles¶

  1. Root re-exports for common types: Frequently used stable types
  2. Module organization by category: processors, stores, resolvers, sinks
  3. Presets module: "Batteries included" factories for common configurations
  4. Visibility control: pub(crate) for workspace-internal items

Testing Strategy¶

  • Unit tests: Within each crate (#[cfg(test)] modules)
  • Integration tests: In tests/integration/
  • End-to-end tests: In tests/e2e/
  • Benchmarks: In benches/
  • Test fixtures: In tests/fixtures/

Build and Release¶

Debug Build¶

cargo build

Release Build¶

cargo build --release

Running Tests¶

cargo test --workspace

Running Service¶

cargo run -p service-http
